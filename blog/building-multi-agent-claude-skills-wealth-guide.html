Building Multi-Agent Claude Code Skills

Most Claude Code skills are one prompt in a markdown file. That works for single-turn tasks. For anything ambitious — a tax tool, a legal intake, a financial planner — you'll collapse into a tangle of conditionals inside one system prompt. This is the architecture that gets you out of that hole, walked through with a working open-source example.

By Srivatsa Kasagar · April 17, 2026 · 13 min read
Share: LinkedIn X

The Problem Every Skill-Builder Hits

You start with a Claude Code skill that does something genuinely useful. Summarize a codebase. Walk a user through a tax filing. Review a contract. You write a single SKILL.md with a good system prompt and some trigger phrases. It works. You ship it.

Then the scope grows. You want the skill to ask questions, branch on answers, read external reference material, cross-check its reasoning, and produce a structured output a non-technical user can actually act on. You try to do all of that inside the one prompt. By version four you have a 3,000-word system prompt with numbered steps, nested conditionals, and a table of contents. By version seven you're debugging why the output occasionally drops sections. By version ten you've accepted that any interesting change risks breaking something unrelated.

This is the failure mode. A real product is not a single prompt. It's a pipeline. Different parts of the problem need different context, different models, and sometimes parallel execution. The skill format supports this, and Claude Code's subagent primitive is exactly the tool — but there is almost no written guidance on how to structure it.

I recently open-sourced wealth-guide, a Claude Code skill that builds a CFP-quality financial plan for US, Canadian, and Indian users. Six specialist subagents. A curated knowledge base. Templated output. Local state. This post is the architecture walkthrough I wish I'd had before building it.

Why a Single Prompt Collapses

The naive first pass at wealth-guide was exactly the thing I'm warning against — one large skill file that read a user's answers, walked through a mental checklist, and generated a plan. Here's an abbreviated version of what that looked like:

---
name: wealth-guide
description: Build a personalized financial plan.
---

# Wealth Guide

Ask the user about their income, expenses, savings, debt, and goals.
Then:

1. Assess their financial health and identify what's broken.
2. Consider their country's tax system and retirement accounts.
3. Look at current market conditions.
4. Generate 3-5 investment strategies with specific fund picks.
5. Stress-test each strategy against market crashes.
6. Produce a prioritized action plan with deadlines.

Country-specific rules:
- If US: 401(k), IRA, Roth, HSA. Use Vanguard tickers.
- If Canada: RRSP, TFSA, FHSA. Consider Smith Manoeuvre.
- If India: old vs new regime, Section 80C layering, SIPs in Nifty 50.

Use current tax year limits. Do not recommend products you are not
sure about. Be specific with dollar amounts. ...

Every run of this produced something. Some runs produced something decent. None produced the same output twice given the same inputs. Section 4 occasionally skipped fund tickers. Section 5 sometimes became three sentences of hedging. Tax limits drifted year-to-year because the model was reconstructing them from training data rather than looking them up. There was no traceable reasoning — when a number was wrong, you could not tell whether the diagnosis was wrong, the research was wrong, or the synthesis had just wandered.

A real wealth management firm does not have one person do all of this. It has an intake analyst, a research team, a market desk, a strategist, a risk officer, and a planner. Each has their own context and their own specialization, and they check each other's work. The skill needs the same shape.

The Mental Model: Mirror the Firm

Once I stopped trying to fit everything into one prompt and started thinking of the skill as a small firm, the design got much easier. The SKILL.md stops being the worker. It becomes the orchestrator — the partner who decides who does what, in what order, and passes findings downstream.

In wealth-guide, the partner dispatches six specialists across three phases:

Phase Specialist What they do
Phase 1 (parallel) financial-diagnostician Scores savings rate, debt-to-income, emergency fund coverage. Benchmarks against country + income bracket.
Phase 1 (parallel) knowledge-advisor Reads the full knowledge base, matches strategies and tax rules to the user's profile, runs web search for current news.
Phase 1 (parallel) market-context-analyzer Pulls live interest rates, index performance, inflation, gold prices via web search.
Phase 2 wealth-strategist Synthesizes phase 1 into 3–5 ranked strategies with fund tickers, dollar amounts, tax-optimized account placement.
Phase 2 risk-reward-evaluator Stress-tests each strategy (40% crash, job loss, rate shock). Assigns verdicts: recommended, caution, not suitable.
Phase 3 action-plan-generator Synthesizes everything into a 12-section roadmap using a template with placeholder slots.

That's the architecture. The rest of this post is how each piece is actually built.

Step 1 — Decompose into Subagents

Claude Code ships a Task tool that launches a subagent with its own context, its own system prompt, and optionally its own model. The subagent runs, returns a result, and the main conversation continues. This is the primitive. What makes it useful for skills is that each subagent definition lives in its own file and you reference it by subagent_type.

Here's the skeleton of a single specialist call from wealth-guide's SKILL.md:

Task(
    subagent_type="financial-diagnostician",
    model="claude-sonnet-4-5-20250929",
    description="Diagnose financial health",
    prompt=f"""
    Analyze this user's financial health.

    Profile:
    - Country: {country}
    - Age: {age}
    - Annual income: {income}
    - Monthly expenses: {expense}
    - Savings: {savings}
    - Investments: {investments}
    - Debt: {debt_summary}
    - Risk tolerance: {risk}

    Score the following 0-100 and return JSON:
    - health_score (overall)
    - savings_rate
    - debt_to_income
    - emergency_fund_months
    - spending_efficiency

    For each score, explain what's working and what's bleeding money.
    Benchmark against people in the same country + income bracket.
    """,
)

Each subagent has exactly one job. The diagnostician scores the financial position. It does not write strategy. It does not look at markets. If you ask it to do more, you get back the same kind of unfocused output you were trying to escape. The discipline is: one agent, one responsibility, one structured output.

The orchestrator reads the result, extracts the structured fields, and hands them to whichever agent needs them next. The main SKILL.md file stops being a giant prompt and becomes a coordination script — Python that reads config, collects interview answers, fires Task calls in the right order, and parses results.

Step 2 — Parallelize Phase 1

The diagnostician, the knowledge advisor, and the market analyzer don't need each other's output. They all read the user's profile and produce independent findings. Running them sequentially is wasted wall-clock time.

Claude Code runs Task calls in parallel when they appear in a single response block. Put three Task invocations in one message — they fire concurrently. Put them in three separate messages — they run one after the other. This is the single most important performance lever in the whole architecture.

# All three Task calls in one block = parallel execution
Task(
    subagent_type="financial-diagnostician",
    model="claude-sonnet-4-5-20250929",
    prompt=f"Analyze financial health for profile: {profile_json}",
)

Task(
    subagent_type="knowledge-advisor",
    model="claude-sonnet-4-5-20250929",
    prompt=f"""Read every file in {kb_dir} and {wf_dir}.
    Match strategies to this profile: {profile_json}.
    Run web search for current {country} financial news.
    Return matched strategies as JSON.""",
)

Task(
    subagent_type="market-context-analyzer",
    model="claude-sonnet-4-5-20250929",
    prompt=f"Pull current rates, indices, inflation for {country}.",
)

Wall-clock time drops from about 3 minutes sequential to around 60–75 seconds parallel, bounded by the slowest of the three. For a user waiting on a plan, that's the difference between "this is slow" and "this is responsive."

⚠️ Ordering matters more than it looks

Any code in SKILL.md that reads a phase 1 result must appear after all three Task blocks, not between them. Putting a print or an assignment between two Task calls is enough to serialize them. Claude Code batches only the adjacent calls in a single response. This is easy to miss during refactoring and the only symptom is that things are slow again.

Step 3 — Ground in a Knowledge Base

Subagent reasoning is only as good as what it reads. If the knowledge-advisor's "knowledge" is whatever the base model remembers from its training data, you get yesterday's tax limits, outdated contribution caps, and product recommendations that were fine three years ago. For anything regulated — tax, legal, medical, financial — that is a liability.

wealth-guide ships a curated knowledge base: seven markdown files, roughly 2,300 lines total, with country-scoped sections for US, Canada, and India.

knowledge/
├── career-income-growth.md
├── index-investing.md
├── money-fundamentals.md
├── real-estate-investing.md
├── retirement-planning.md
├── side-hustles.md
└── tax-optimization.md

workflows/
├── debt-freedom.md
├── first-investment.md
├── side-hustle-launch.md
└── wealth-building.md

Each file contains real numbers — current contribution limits, tax brackets, specific fund tickers with expense ratios, APR ranges, methodology citations, and step-by-step decision frameworks. A snippet of tax-optimization.md:

## Canada - Tax-Advantaged Accounts (2024)

### RRSP (Registered Retirement Savings Plan)
- Contribution room: 18% of earned income, capped at $31,560
- Deduction reduces taxable income in the year of contribution
- Withdrawals fully taxable; avoid withdrawing before retirement
- Spousal RRSP: income-split in retirement if partner in lower bracket

### TFSA (Tax-Free Savings Account)
- 2024 contribution: $7,000. Lifetime room if 18+ since 2009: $95,000
- Growth and withdrawals tax-free; room restored next calendar year
- Hold highest-growth assets here (equities, not GICs)

### FHSA (First Home Savings Account)
- Annual limit: $8,000. Lifetime: $40,000
- Combines RRSP-style deduction with TFSA-style tax-free growth
- Unused room carries forward, up to $8,000 per year

The knowledge-advisor is instructed to quote from this base when matching strategies, so every recommendation in the final plan traces back to a specific methodology and a specific number. When a user asks "where did 18% come from," you can point at a line in a file. That is auditability. A single-prompt skill cannot give you that.

Step 4 — Deterministic Output with Templates

Left to its own devices, an LLM will write prose that varies in structure every run. For a product where users expect the same 12 sections in the same order every time — executive summary, financial position, projections, strategy blueprint, debt plan, tax plan, cash flow, risk assessment, action plan, market context, learning path, key dates — prose is not good enough.

The pattern: a 232-line roadmap-template.md with {PLACEHOLDER} slots, and a final-stage agent whose job is to fill the placeholders. Excerpted:

# Your Financial Roadmap

## 1. Executive Summary

{EXECUTIVE_SUMMARY_TABLE}

{EXECUTIVE_SUMMARY_NARRATIVE}

## 2. Financial Position

Health Score: {HEALTH_SCORE}/100 ({HEALTH_BAND})

{NET_WORTH_BREAKDOWN}

{CASH_FLOW_SUMMARY}

## 3. Wealth Projections (10-Year)

{PROJECTIONS_TABLE}

## 4. Strategy Blueprint

{STRATEGY_1_BLOCK}
{STRATEGY_2_BLOCK}
{STRATEGY_3_BLOCK}

...

The action-plan-generator is instructed to produce content for each placeholder separately, using the diagnostician's scores, the strategist's ranked strategies, and the risk-evaluator's verdicts as inputs. The template enforces the shape. The agent enforces the content. Structure stops being a generation concern and becomes a compile-time guarantee.

The same pattern ports cleanly. If you were building a legal intake skill, you'd have a template with {CLIENT_SUMMARY}, {JURISDICTION_ANALYSIS}, {RECOMMENDED_FORMS}, {DEADLINES_TABLE}. If you were building a code review skill, you'd template {ARCHITECTURE_NOTES}, {SECURITY_FINDINGS}, {PERFORMANCE_FLAGS}. The mechanic doesn't change.

Step 5 — Route Opus vs Sonnet by Cognitive Load

Not every agent needs the same model. Scoring a user's savings rate is a mechanical task — Sonnet is fast, cheap, and entirely sufficient. Synthesizing a ranked strategy set that optimizes across tax, time horizon, and country-specific account mechanics is genuinely hard reasoning — worth Opus.

The routing in wealth-guide:

Agent Model Why
financial-diagnostician Sonnet 4.5 Pattern-match scoring against benchmarks. Fast, cheap, good enough.
knowledge-advisor Sonnet 4.5 Reading + matching. Structured task, no multi-step inference.
market-context-analyzer Sonnet 4.5 Web search + summary. Zero creative reasoning needed.
wealth-strategist Opus 4.6 Multi-constraint optimization across tax, time, country, risk, and instrument availability. This is the hard thinking.
risk-reward-evaluator Sonnet 4.5 Scoring and verdict-stamping. Sonnet follows a clear rubric well.
action-plan-generator Opus 4.6 Synthesizing everything upstream into a coherent 12-section plan with internal consistency. Opus keeps the threads together.

Four Sonnet runs, two Opus runs. The Opus agents get 120-second timeouts in config/agent-config.yaml because serious reasoning takes time. The Sonnet agents get 60. The economics work out to a single full roadmap costing a small fraction of what Opus-everywhere would cost, with no visible quality drop on the parts that don't need Opus.

Step 6 — Local State with SQLite

LLMs are stateless between runs. A multi-session product isn't. Users want to come back tomorrow and not re-answer ten interview questions. They also want their financial data stored somewhere they control, not in an API provider's logs.

wealth-guide stores profile data in a local SQLite file at ~/.claude/skills/wealth-guide/data/profiles.db, with 0600 permissions set on creation. Two tables:

CREATE TABLE profiles (
    id INTEGER PRIMARY KEY,
    country TEXT,
    age INTEGER,
    annual_income REAL,
    monthly_expense REAL,
    savings TEXT,
    investment_assets TEXT,
    debt_credit_card TEXT,
    debt_personal_student TEXT,
    debt_mortgage TEXT,
    risk_tolerance TEXT,
    experience TEXT,
    goal TEXT,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE session_history (
    id INTEGER PRIMARY KEY,
    profile_id INTEGER,
    strategies_matched TEXT,
    roadmap_path TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY(profile_id) REFERENCES profiles(id)
);

On startup the orchestrator checks for an existing profile. Under 24 hours old, it offers to reuse. Between 1 and 30 days, it offers a partial refresh (update income/expenses only). Past 30 days, fresh interview. The thresholds are in agent-config.yaml so you can tune without code changes.

⚠️ Never send raw user financial inputs to an external API

The skill asks the user questions like "annual income" and "total debt." The subagents don't receive those raw numbers. They receive midpoints of bracket ranges — the user picked "$100k-$150k", the agent sees "$125k". This keeps individual financial data out of the model's context and out of any inference provider's logs. For anything regulated, bracket-midpoint aggregation is the minimum standard. Raw-value passthrough is the single most common PII mistake I see in AI products, and it's almost always avoidable.

Putting It Together: wealth-guide

The full skill is open-source and MIT licensed. The directory layout:

wealth-guide/
├── SKILL.md                   # Orchestrator (1,802 lines)
├── config/
│   ├── agent-config.yaml      # Timeouts, thresholds, retry
│   └── init_db.py             # SQLite schema
├── knowledge/                 # 7 curated MD files, US/CA/IN
├── workflows/                 # 4 step-by-step playbooks
├── templates/
│   └── roadmap-template.md    # 232-line output template
├── scripts/
│   └── md_to_html.py          # Print-ready HTML export with charts
├── data/                      # SQLite profile DB (gitignored)
├── roadmaps/                  # Generated plans (gitignored)
└── README.md

Install is one symlink — no pip install, no virtualenv, no dependencies beyond Python standard library for the SQLite piece:

git clone https://github.com/caseonix/wealth-guide.git

mkdir -p ~/.claude/skills
ln -s "$(pwd)/wealth-guide" ~/.claude/skills/wealth-guide

# In any Claude Code session:
wealth guide

The interview runs, the six specialists work the case, and a 12-section markdown roadmap lands in ~/.claude/skills/wealth-guide/roadmaps/ along with a print-ready HTML version with charts — a net worth donut, a wealth projection line chart, a cash flow waterfall, and a health score gauge. Cmd+P in the browser gives you a clean PDF.

Build vs Custom GPT vs LangChain vs Claude Code Skill

There are other ways to build something like this. They trade off differently.

Approach Time to ship Parallel agents Local state Forkable Runs where user works
Custom GPT Hours No No Partial ChatGPT only
LangChain + FastAPI Weeks Yes, you wire it Yes, you wire it Yes Your web app
Claude Agent SDK Days Yes, built in Yes, you wire it Yes Your app / CLI
Claude Code skill (this post) Days Yes, built in Yes, local SQLite Yes, MIT repo Developer's own terminal

The skill approach wins specifically when your user is already a Claude Code user — developers, power users, people comfortable in a terminal. It loses when the user wants a URL to hit from a browser. For wealth-guide's audience (technical people curious about their own finances), skills were obviously the right substrate. For a consumer product with signup, use the Agent SDK and wrap it in a web app.

Sibling Project: Consul (For When You Want the Browser Version)

The same architectural principles port cleanly to a hosted web app. I know, because I built that version too — it's called Consul, and it's what I reach for when I want to do this kind of analysis on my own portfolio without standing up a terminal.

Consul is narrower in scope than wealth-guide — it's specifically a portfolio narrative tool, not a full life-stage financial planner. You drop in your positions (manually, CSV, or brokerage import), it enriches them with live market data (prices, sectors, beta, dividend yields, expense ratios), and Claude writes a plain-English strategic assessment: concentration risks, hidden exposures, fee drag, and actionable next steps. Multi-jurisdiction aware — US, Canada, UK, Australia, with understanding of TFSA, RRSP, IRA, Roth IRA, and taxable account mechanics.

The architectural principle it shares with wealth-guide is what I think matters most for any AI-in-finance product: math is deterministic, AI writes the story. Sharpe ratio, VaR at 95%, beta, HHI concentration, CAGR, 20-year fee drag — all computed in deterministic code before the model sees them. Claude never performs arithmetic; it only explains what the numbers mean for you. Every reasoning step is logged and inspectable. Every action needs explicit human approval. No black boxes.

Consul is live at consol-app.pages.dev. It is closed-source and actively developed — a deliberate contrast to wealth-guide, which is MIT-licensed and frozen at a clean reference-implementation state. Think of them as a matched pair: fork wealth-guide when you want to build your own skill; use Consul when you want the analysis without the build.

📈 If you're deciding between them

wealth-guide is for multi-year life planning (goals, debt payoff, savings runway, country-specific tax optimization, 12-section roadmap). Consul is for portfolio-level analysis of what you already hold (risk, concentration, fees, narrative). They overlap only at the edges.

Practical Notes from Building This

A few things I'd have wanted to know earlier:

  • Parallel Task blocks are fragile to refactor. The rule — no non-Task code between Task calls you want parallel — holds rigidly. A well-meaning print("Dispatching...") in the middle of a block will silently serialize the agents. Comment-only lines are fine; any executable statement is not.
  • Opus agents need 120-second timeouts, not 60. The strategist and planner produce 800–1,500 tokens of tightly-structured output after multi-step reasoning. Sonnet will cap at 60s comfortably; Opus needs headroom. Get this wrong and you'll see sporadic truncation on your hardest-thinking agents.
  • Country context must propagate through every agent prompt. The user picks their country once, in screen 1 of the interview. If the diagnostician gets the country but the strategist doesn't, you get a US-style recommendation for an Indian user. Pass it explicitly into every Task prompt; don't rely on the orchestrator to "remember."
  • Bracket ranges, midpoints, never raw. The interview uses four-option multiple-choice buckets (income, savings, debt). The orchestrator converts buckets to midpoints before any agent sees them. This is both better UX (no mental math) and better privacy.
  • Put every tunable in YAML. Timeouts, retry counts, cache windows, skill-level thresholds. Anything that might want changing without a code edit goes in config/agent-config.yaml. Future you — or anyone forking the skill — will thank you.
  • The knowledge base is the highest-leverage file to maintain. New tax year? Update the limits in one markdown file. New regulatory change? Add a section. No agent prompt needs to change. The plumbing is the expensive part to build once; the content is the expensive part to keep fresh.
✅ The short version

Stop cramming ambitious skills into one prompt. Decompose into subagents with single responsibilities. Parallelize the independent ones in a single response block. Ground reasoning in a curated knowledge base with real numbers. Template the output so structure is a compile-time guarantee. Route Opus only where reasoning is genuinely hard. Keep state in local SQLite with 0600 permissions. Never send raw user values across the API boundary. wealth-guide is the full working reference — MIT licensed, fork it for your own domain.
