LocalMind RAG — the pipeline

Apr 22, 2026 · Working note — will evolve
1. Pipeline 2. Quality improvements → 3. NLP layer →
§ Live app localmind.caseonix.ca ↗

How I wired retrieval-augmented generation end-to-end in LocalMind: chunking, embedding, vector index, retrieval, generation. Everything runs on the Cloudflare edge — Workers AI for inference, Vectorize for ANN, D1 for the canonical chunk text, and R2 for the original blob. I use teamId as the Vectorize namespace on every read and write to keep tenants isolated.

Ingest pipeline (write path)

I put the whole ingest pipeline inside a per-team Durable Object so it can run async, hold rate-limit state, and use alarms to verify the vector index.

Diagram 1 — ingest pipeline inside the per-team Durable Object

Orchestration lives in workers/src/durableObjects/teamState.ts (processDocument). The main twist I added is a dual-text split: D1 stores the redacted-but-otherwise-original chunk that users see in citations, and Vectorize stores a contextually enriched version where Gemma 4 has prepended a 1-2 sentence header that says where the chunk fits in the document (via generateChunkContext). The header improves recall for chunks that would otherwise be too short or too back-referencing to retrieve on their own, and it never shows up in the user-facing citation.

Chunking

The chunker in workers/src/services/document.ts (chunkText) is small and deterministic. It splits the text on paragraph breaks (\n{2,}) and packs whole paragraphs into a buffer until the buffer would go over MAX_CHUNK_TOKENS (512 tokens, approximated as 4 chars/token). When a paragraph won't fit, I flush the buffer and carry a 50-token tail forward as overlap so context isn't cut off in the middle of a thought. If a single paragraph is bigger than the cap on its own, I fall back to a sentence regex and pack sentences instead. No tokenizer call, no LLM — paragraph structure is the main signal, and the token math is conservative on purpose.

Before chunks ever touch D1 or Vectorize, I run them through two PII passes (regex + LLM). The redaction is applied to both the display copy in D1 and the version that gets embedded — privacy was a hard requirement for me, so nothing identifiable leaves the chunk pipeline. The PII layer is detailed in the NLP note.

Embedding

The embedding service in workers/src/services/embedding.ts is small on purpose — a single short module with two functions. It calls @cf/baai/bge-small-en-v1.5 via Workers AI for both the document side (generateEmbeddings) and the query side (generateQueryEmbedding). On the query path I prepend BGE's asymmetric prefix "Represent this sentence for searching relevant passages: ". That's how BGE was trained on the query side, and using it raises cosine similarity for relevant chunks above the noise floor.

In the DO, I batch embeddings at 32, then call Vectorize.upsert in batches of 100. Vectorize is eventually consistent, so once the upsert returns I schedule a Durable Object alarm 90 seconds out. The alarm re-queries the first and last chunk IDs to check they're indexed, retries up to 5 times, and only marks the document ready for search once both ends are visible. If verification fails after the final retry, I flip the document to error so the user knows search won't work for it.

Diagram 2 — retrieval flow inside POST /api/search

The whole flow is in the POST /api/search handler in workers/src/routes/search.ts. Step by step:

  1. Query expansion (expandQuery). Gemma rewrites the user's query into 2-3 variants under a JSON schema. The original is always kept, and I cap variants at QUERY_EXPANSION_MAX = 3. If the LLM call fails I fall back to the original query only.
  2. Multi-vector search. I embed each variant with the BGE query prefix and dispatch all of them in parallel to Vectorize.query, scoped to the team's namespace with topK = 15.
  3. Merge and dedupe. I union results by chunk ID and keep the highest cosine score across variants — so a chunk that matches multiple phrasings gets credit but isn't double-counted.
  4. Hydrate and threshold. I load chunk text from D1 in a single inArray query, join it to document names, and drop anything below MIN_SIMILARITY = 0.3. (BGE-Small puts truly relevant chunks in the 0.35-0.55 band on my corpus, so 0.3 is the tuned floor.)
  5. LLM rerank (rerankChunks). I send the top 15 candidates to Gemma with a strict 1-5 relevance rubric, again JSON-schema constrained. I cut anything below RERANK_MIN_SCORE = 3 and keep RERANK_TOP_K = 7. If fewer than 2 chunks pass, I fall back to the vector ranking.
  6. Context assembly (buildContextFromChunks). I concatenate chunks with [N] filename headers and --- separators, stopping when I hit MAX_CONTEXT_TOKENS = 6000. For team-wide searches I cap each document at MAX_CHUNKS_PER_DOC_TEAM_SEARCH = 2 so one chatty document can't crowd out the others.
  7. Synthesis (synthesizeAnswer). The system prompt binds Gemma to the supplied context, requires [Document: filename] citations, and forbids external knowledge. I walk conversation history newest-first and trim it at MAX_HISTORY_TOKENS = 1500 so long chats don't blow the context window.

Models

  • Embedder: @cf/baai/bge-small-en-v1.5 (BGE-Small, 384-dim) on Workers AI.
  • Generator / reranker / classifier: @cf/google/gemma-4-26b-a4b-it, configurable via WORKERS_AI_MODEL.
  • Vector index: Cloudflare Vectorize, partitioned by teamId namespace.
  • Chunk store: Cloudflare D1 (SQLite) via Drizzle.
  • Blob store: Cloudflare R2.
